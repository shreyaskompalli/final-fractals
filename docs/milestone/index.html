What we Have Accomplished

Unity

Our project needed to be built from scratch in Unity, so we first familiarized ourselves with the various components and functionalities of Unity. In particular, we learned how to code in high-level shader language (HLSL) and use the scene in Unity. Because of the relative difficulty of debugging in HLSL, we’ve defined a custom function to help us identify whether we are running into a bug or not by rendering different colors to the scene.

Ray marcher

Since fractals are quite complex shapes, we decided to use the ray marcher algorithm instead of the ray tracing algorithm we learned in class. To do that, we self-learned how to implement the ray marcher algorithm and used it to render a sphere and a cube. The big idea of ray marcher is that every object is defined using a signed distance function. So, instead of intersecting with shapes multiple times, at every step, we find the closest point from the current position to the object, and then advance that much distance in the direction of the ray until we are close to the object. We check whether we are close to the object by calculating whether the distance is small or not.

HLSL

HLSL (High-Level Shader Language) is the default shader language used in GLSL, and we needed to learn how to use this language in order to get our ray marcher to render on the GPU in real-time. In addition to learning new syntax and semantics, we also had to learn how to pass in information from C# scripts run by the CPU into the shader code so that it could be processed by the GPU. This allowed us to work around the fact that HLSL typically does not have access to Unity scene information.

SDFs

The ray marcher does not use traditional ray-primitive intersection functions like a ray tracer does, so we need to use Signed Distance Functions (SDFs) to tell our ray marcher whether a ray intersects with a primitive. SDFs function similarly to implicit geometry, as they return the closest distance from a ray to the surface of an object. We can leverage the fact that fractals have known SDFs to render very detailed geometries for relatively little computational cost, as each ray will only have to do one call to the SDF per step of the ray march instead of potentially millions of triangle intersection tests in a ray tracer.

Lighting effects & coloring

We assigned Vector4s representing (R, G, B, A) colors to each of the primitives in our scene and pass that data to the ray marcher. When a ray in the ray marcher intersects with a primitive, it fetches the color of that primitive and uses it to compute diffuse shading using Lambert’s law as demonstrated in class. We also added a small ambient shading term in order to make shapes more legible in low light conditions.

Preliminary Results

As of right now, we have learned how to use Unity to render scenes in real time using the HLSL shaders. Currently, we have a functional renderer that is able to generate rays, using our 3-1 code and the camera position, that then traces them using the Ray Marcher Algorithm, taking into account the object SDF’s to test for intersection.


Once we programmed this, we noticed that our sphere was slightly distorted, so we decided to add a debug color as well, since debugging shaders is increasingly difficult due to the lack of print statements. This allowed us to see that the debug colors showed that the rays were being generated correctly.


Our next step was adding color effects for diffuse lighting to the sphere and cube to see if there were any real problems. After doing this, we essentially completed the milestone if not more. We implemented Ray Marching, and can render spheres and cubes to the screen in real time, along with diffuse lighting. These are our preliminary results, however, we have some visual artifacts that we want to investigate further to see if it is an actual problem. The first artifact is that when we move the sphere or cube to the left or right in the scene, it becomes distorted, similar to a fisheye type of distortion effect. We aren’t sure if this is due to the horizontal field of view yet, but we will investigate this further. The second visual artifact is that if we move the sphere into the cube, it looks like there is a black hole effect, which we aren’t sure why would happen.

Reflection of Progress Relative to the Plan

We are actually ahead of schedule, as we planned to have a working ray marcher without lighting effects done by this week and have lighting effects added last week. However, we concluded that lighting effects added a lot to the depth and feel of the geometry in the scene, so we decided to include diffuse and ambient lighting before the date in our plan.
